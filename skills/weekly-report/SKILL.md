# 周报生成器

把用户描述的本周工作内容整理成一份结构清晰、语气亲和的中文周报，保存为 Markdown 文件。

## 周报结构

周报由两部分组成：

### 1. 开头总结

用 1-2 句话概括本周工作重点，语气像跟同事聊天一样自然。不要用"尊敬的领导"这种正式称呼，也不要太随意。目标是让读者一眼就知道这周主要忙了什么。

好的例子：
- "这周主要在推进用户系统的重构，同时处理了几个线上问题。"
- "本周重心放在 Q2 需求评审和技术方案设计上，另外把积压的两个 bug 修掉了。"

避免的风格：
- 太正式："本周工作汇报如下，请各位领导审阅。"
- 太随意："这周摸了一周鱼哈哈。"
- 太空洞："本周完成了多项工作任务。"

### 2. 任务表格

用 Markdown 表格呈现，三列：

| 任务 | 进度 | 备注 |
|------|------|------|
| 具体任务描述 | 进度状态 | 补充说明 |

**任务列**：简洁描述做了什么，不需要太详细但要能让人看懂。如果用户给的描述太笼统，适当补充细节让它更有信息量。

**进度列**：用文字描述，常用的有：
- 已完成
- 进行中（可以加百分比辅助，如"进行中，约 70%"）
- 待开始
- 已上线
- 已提测
- 已评审

根据用户描述的实际情况选择最合适的状态，不要生搬硬套。

**备注列**：放补充信息，比如：
- 遇到的问题或阻塞
- 下周计划
- 需要协助的事项
- 关联的人或团队

如果没有特别需要说明的，备注列可以留空（用 `-` 填充）。

## 信息整理原则

用户可能会很随意地描述工作内容，你需要做的是：

1. **归类合并**：相关的小事项合并成一个任务，避免表格太碎。比如"修了登录页的样式""调了注册页的间距"可以合并成"修复登录/注册页面样式问题"。
2. **补充推断**：如果用户说"做完了"，进度就是"已完成"；说"还在搞"就是"进行中"；说"刚开始看"就是"进行中，约 10%"。
3. **合理排序**：按重要程度或工作量排序，重要的任务放前面。
4. **保持原意**：不要添加用户没提到的工作内容，整理≠编造。

## 文件保存

保存为 `weekly-report-YYYY-MM-DD.md`，日期用当天日期。文件保存在当前工作目录下。

如果当前目录下已经存在同名文件，提醒用户是否覆盖。

## 完整示例

用户输入：
> 这周把商品详情页的缓存优化做完了，上线了。还在搞订单导出功能，大概做了一半。另外帮前端团队 review 了几个 PR，还处理了一个线上告警。

生成的周报：

```markdown
# 周报 - 2026-04-24

这周主要完成了商品详情页的缓存优化并成功上线，订单导出功能也在稳步推进中。

| 任务 | 进度 | 备注 |
|------|------|------|
| 商品详情页缓存优化 | 已上线 | 优化完成并部署上线 |
| 订单导出功能开发 | 进行中，约 50% | 预计下周完成 |
| 协助前端团队 Code Review | 已完成 | 审核了多个 PR |
| 处理线上告警 | 已完成 | - |
```
